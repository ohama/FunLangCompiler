**FIX-03: I64->I1 coercion in And/Or/WhileExpr**

Copy the coercion pattern from the `If` case (lines 809-814). The pattern is:
```fsharp
let (i1Val, coerceOps) =
    if val.Type = I64 then
        let zeroVal = { Name = freshName env; Type = I64 }
        let boolVal = { Name = freshName env; Type = I1  }
        (boolVal, [ArithConstantOp(zeroVal, 0L); ArithCmpIOp(boolVal, "ne", val, zeroVal)])
    else (val, [])
```

Apply to `And` case (~line 834):
- Coerce `leftVal` to `i1LeftVal` after elaborating lhsExpr
- Coerce `rightVal` to `i1RightVal` after elaborating rhsExpr
- Use `i1LeftVal` in `CfCondBrOp` (both branch condition AND false-path merge arg)
- Use `i1RightVal` in `CfBrOp` to merge block
- Include `coerceLeftOps` in the returned ops (after leftOps)
- Include `coerceRightOps` in the evalRight block body (after rightOps, before CfBrOp)

Apply to `Or` case (~line 845): same pattern, but note Or has REVERSED branch targets in CfCondBrOp (true goes to merge, false goes to evalRight). So:
- Coerce `leftVal` -> `i1LeftVal`, `rightVal` -> `i1RightVal`
- Use `i1LeftVal` in CfCondBrOp condition AND true-path merge arg
- Use `i1RightVal` in CfBrOp to merge

Apply to `WhileExpr` case (~line 2947): TWO condition elaborations need coercion:
1. `condVal` (line 2956, header block condition) - coerce before use in CfCondBrOp at line 2986
2. `condVal2` (line 2962, back-edge condition) - coerce before use in CfCondBrOp at line 2964
- Insert coercion ops into the respective block bodies alongside the condOps/condOps2

**IMPORTANT:** Do NOT change any other logic. The fixes are additive (inserting blocksAfterBind capture, inserting coercion ops). Do not refactor surrounding code.
