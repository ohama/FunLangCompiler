**Part A: Modify Program.fs for error categorization (CAT-01/CAT-02)**

In the `main` function (line ~126), replace the single try/catch with structured error handling that distinguishes parse vs elaboration errors.

Current code (lines 126-199):
```fsharp
try
    ...
    let ast = parseProgram combinedSrc inputPath
    ...
    let mlirMod = Elaboration.elaborateProgram expandedAst
    match Pipeline.compile mlirMod outputPath with
    ...
with ex ->
    eprintfn "Error: %s" ex.Message
    1
```

Change to separate try/catch blocks so parse errors get `[Parse]` prefix:

```fsharp
// Step 1: Parse
let expandedAst =
    try
        let ast = parseProgram combinedSrc inputPath
        // ... expand imports (keep existing code) ...
        expandedAst
    with ex ->
        eprintfn "[Parse] %s" ex.Message
        exit 1  // use exit to avoid nesting

// Step 2: Elaborate
let mlirMod =
    try
        Elaboration.elaborateProgram expandedAst
    with ex ->
        // Elaboration errors already have [Elaboration] prefix from failWithSpan
        eprintfn "%s" ex.Message
        exit 1

// Step 3: Compile
match Pipeline.compile mlirMod outputPath with
| Ok () -> 0
| Error (Pipeline.MlirOptFailed (code, err, mlirFile)) ->
    eprintfn "[Compile] mlir-opt failed (exit %d):\n%s\nMLIR file preserved: %s" code err mlirFile
    1
| Error (Pipeline.TranslateFailed (code, err, mlirFile)) ->
    eprintfn "[Compile] mlir-translate failed (exit %d):\n%s\nMLIR file preserved: %s" code err mlirFile
    1
| Error (Pipeline.ClangFailed (code, err)) ->
    eprintfn "[Compile] clang failed (exit %d):\n%s" code err
    1
```

IMPORTANT IMPLEMENTATION NOTES:
- Do NOT use `System.Environment.Exit` — it's not idiomatic F#. Instead, restructure the logic. One clean approach: keep a single outer try/catch, but detect elaboration errors by checking if `ex.Message` starts with `[Elaboration]`. For parse errors, catch the parseProgram call separately and re-raise with a `[Parse]` prefix. Or: use a custom exception type `ElaborationException` raised by failWithSpan — but that's more invasive.
- SIMPLEST approach: Keep the single try/catch at line 197, but change the handler:
  ```fsharp
  with ex ->
      if ex.Message.StartsWith("[Elaboration]") then
          eprintfn "%s" ex.Message
      else
          eprintfn "[Parse] %s" ex.Message
      1
  ```
  This works because:
  - Elaboration errors from failWithSpan now start with `[Elaboration]` (from Task 1)
  - Parse errors (from FsLexYacc) do NOT start with `[Elaboration]`
  - Pipeline compile errors are already handled by the `match Pipeline.compile` above the catch
  
- For the compile errors, just add `[Compile]` prefix to the existing eprintfn calls (lines 189, 191, 194).
