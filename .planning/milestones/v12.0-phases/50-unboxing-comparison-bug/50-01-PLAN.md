<objective>
Fix the unboxing comparison bug where ordinal comparison operators (>, <, >=, <=) fail on boxed Ptr operands by adding Ptr-to-I64 coercion using the existing coerceToI64 helper.

Purpose: When isPtrParamBody coerces a lambda parameter to Ptr (e.g., `fun x -> if x > 2 then Some x else None`), comparison operators must unbox before emitting arith.cmpi. Currently they pass Ptr directly, causing MLIR type mismatch.
Output: Working comparison predicates on boxed integer lists, verified by E2E tests.
</objective>

In `elaborateExpr` in Elaboration.fs, modify the four ordinal comparison cases (LessThan ~line 1081, GreaterThan ~line 1086, LessEqual ~line 1091, GreaterEqual ~line 1096) to apply `coerceToI64` to both operands before emitting ArithCmpIOp.

For each of the four cases, replace the current pattern:

```fsharp
| GreaterThan (lhs, rhs, _) ->
    let (lv, lops) = elaborateExpr env lhs
    let (rv, rops) = elaborateExpr env rhs
    let result = { Name = freshName env; Type = I1 }
    (result, lops @ rops @ [ArithCmpIOp(result, "sgt", lv, rv)])
```

With:

```fsharp
| GreaterThan (lhs, rhs, _) ->
    let (lv, lops) = elaborateExpr env lhs
    let (rv, rops) = elaborateExpr env rhs
    let (lv64, lCoerce) = coerceToI64 env lv
    let (rv64, rCoerce) = coerceToI64 env rv
    let result = { Name = freshName env; Type = I1 }
    (result, lops @ rops @ lCoerce @ rCoerce @ [ArithCmpIOp(result, "sgt", lv64, rv64)])
```

Apply this same pattern to all four:
- LessThan: predicate "slt"
- GreaterThan: predicate "sgt"
- LessEqual: predicate "sle"
- GreaterEqual: predicate "sge"

IMPORTANT: Do NOT modify Equal or NotEqual cases. They already handle Ptr correctly via strcmp for string comparison. Adding coercion there would break string equality.

IMPORTANT: The op ordering must be `lops @ rops @ lCoerce @ rCoerce @ [ArithCmpIOp(...)]` — coercion ops come AFTER both operands are elaborated.
