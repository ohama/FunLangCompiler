<objective>
Fix SSA scope violation in 3+ argument curried functions (Issue #4).

Purpose: The 2-lambda pattern in Elaboration.fs incorrectly handles 3+ arg curried functions. When innerBody is itself a Lambda, the standalone Lambda path generates SSA values during inner body elaboration. The maker function (a separate func.func) then references these SSA values from env.Vars, but they belong to a different function scope — causing MLIR "definition of SSA value" errors. The fix adds a guard to the 2-lambda pattern so 3+ lambda chains fall through to the general Let path, which correctly compiles each lambda layer as a nested closure.

Output: Patched Elaboration.fs + new E2E test confirming 3-arg curried function compiles and runs correctly.
</objective>

<task type="auto">
  <name>Task 1: Add guard to 2-lambda pattern to reject 3+ lambda innerBody</name>
  <files>src/FunLangCompiler.Compiler/Elaboration.fs</files>
  <action>
Add a `when` guard to the 2-lambda pattern match arm at line 713.

Current pattern (line 713):
```fsharp
| Let (name, StripAnnot (Lambda (outerParam, StripAnnot (Lambda (innerParam, innerBody, _)), _)), inExpr, letSpan) ->
```

Change to:
```fsharp
| Let (name, StripAnnot (Lambda (outerParam, StripAnnot (Lambda (innerParam, innerBody, _)), _)), inExpr, letSpan)
    when (match stripAnnot innerBody with Lambda _ -> false | _ -> true) ->
```

This single-line guard addition prevents 3+ arg curried functions (where innerBody is itself a Lambda) from matching the 2-lambda pattern. They fall through to the general `Let` pattern at line 953, which calls `elaborateExpr env bindExpr` on the full `Lambda(a, Lambda(b, Lambda(c, body)))`. The standalone Lambda path (line ~3131) handles each lambda layer as a nested closure — each layer is a separate func with correct SSA scope.

WHY this works: The general Let path treats the entire lambda chain as a value. `elaborateExpr` on a Lambda creates an inline closure struct with its own isolated function body. Each nested Lambda recurses, creating another closure inside the previous closure's function body. No SSA values cross function boundaries because each closure function has its own env with its own counter.

WHY NOT a 3-lambda pattern: Adding a dedicated 3-lambda pattern would be more complex and only handle exactly 3 args. The general Let path handles N-ary lambda chains correctly (though without the KnownFuncs optimization). The out-of-scope note in REQUIREMENTS.md confirms: "4+ arg curried function 최적화 — 3-arg가 올바르게 동작하면 4+도 자동 해결 (recursive Lambda 처리)".

DO NOT modify the 1-lambda pattern at line 893 — it correctly rejects multi-lambda and has no SSA scope issues.
DO NOT modify the standalone Lambda path (line ~3131) — it is correct within its own scope.
DO NOT touch the maker capture store logic (lines 818-838) — the fix prevents it from being reached for 3+ lambda.
