# Roadmap Archive: v2.0 Data Types & Pattern Matching

**Status:** SHIPPED
**Phases:** 7-11
**Completed:** 2026-03-26

## Milestone Goal

Extend the compiler with heap-allocated types (string, tuple, list) and pattern matching, backed by Boehm GC runtime. Every heap allocation goes through `GC_malloc`. After this milestone, programs using strings, tuples, lists, and `match` expressions compile and produce correct results.

  3. A program that returns a closure from a function and applies it after the defining stack frame has returned executes correctly (confirms heap closure environments do not become dangling)
  4. `print "hello"` and `println "world"` compile and write the expected strings to stdout in a FsLit E2E test

### Phase 11: Pattern Matching
**Goal**: The `match` expression compiles to a sequential `cf.cond_br` decision chain that handles all v2 pattern types (constant, wildcard, variable, string, empty list, cons, tuple); a non-exhaustive match always has a `@lang_match_failure` fallback block so undefined behavior is impossible
**Depends on**: Phases 8, 9, 10
**Requirements**: PAT-01, PAT-02, PAT-03, PAT-04, PAT-05
**Success Criteria** (what must be TRUE):
  1. `match 42 with | 0 -> "zero" | 42 -> "answer" | _ -> "other"` compiles and the binary prints `"answer"`, confirming constant int pattern and wildcard pattern both work
  2. `let rec sum lst = match lst with | [] -> 0 | h :: t -> h + sum t in sum [1; 2; 3]` compiles and exits with 6, confirming `EmptyListPat` null check and `ConsPat` destructuring work together with recursive calls
  3. `match (1, 2) with | (a, b) -> a + b` compiles and exits with 3, confirming `TuplePat` GEP-load field extraction in match works
  4. `match "hello" with | "hello" -> 1 | _ -> 0` compiles and exits with 1, confirming `strcmp`-based string constant pattern comparison works
  5. A program with a non-exhaustive match on integers (no default arm) links and calls `@lang_match_failure` at runtime when no arm matches â€” the process exits non-zero rather than producing undefined behavior
**Plans**: 2 plans

## Milestone Summary

**Key Decisions:**
- Boehm GC (libgc) â€” conservative collector, no IR changes, `-lgc` link only
- GC_INIT() emitted unconditionally as first op in @main
- All heap allocation through GC_malloc without exception
- Closure environments migrated from llvm.alloca to GC_malloc before any heap type codegen
- String layout: {i64 length, ptr data} two-field heap struct
- EmptyList = llvm.mlir.zero (null ptr); Cons = GC_malloc(16) two-pointer cons cell
- Match compiles to sequential cf.cond_br chain; always emit @lang_match_failure terminal
- GC symbol is @GC_init (lowercase i) â€” GC_INIT is a C macro
- lang_runtime.c compiled per-invocation to temp .runtime.o
- testPattern returns 4-tuple (condOpt, testOps, bodySetupOps, bindEnv) for Jacobs decision tree

**Issues Resolved:**
- LlvmAllocaOp migration â€” only one usage site in App dispatch; fully migrated in 07-02
- macOS -L flag for bdw-gc added in 07-01 via RuntimeInformation.IsOSPlatform
- ArithExtuIOp added to promote I1 to I64 before C ABI calls
- FsLit tests must be single-line â€” FunLang parser does not accept newlines
