  <files>src/FunLangCompiler.Compiler/Elaboration.fs</files>
  <action>
Read Elaboration.fs. The current `Equal` and `NotEqual` cases always emit `ArithCmpIOp`. We need
to dispatch based on the elaborated type of the LHS operand.

Replace the `Equal` case in `elaborateExpr` with a type-dispatching version:

```fsharp
| Equal (lhs, rhs, _) ->
    let (lv, lops) = elaborateExpr env lhs
    let (rv, rops) = elaborateExpr env rhs
    if lv.Type = Ptr then
        // String equality via strcmp
        let lDataPtr  = { Name = freshName env; Type = Ptr }
        let lData     = { Name = freshName env; Type = Ptr }
        let rDataPtr  = { Name = freshName env; Type = Ptr }
        let rData     = { Name = freshName env; Type = Ptr }
        let cmpResult = { Name = freshName env; Type = I32 }
        let zero32    = { Name = freshName env; Type = I32 }
        let boolResult = { Name = freshName env; Type = I1 }
        let ops = [
            LlvmGEPStructOp(lDataPtr, lv, 1)
            LlvmLoadOp(lData, lDataPtr)
            LlvmGEPStructOp(rDataPtr, rv, 1)
            LlvmLoadOp(rData, rDataPtr)
            LlvmCallOp(cmpResult, "@strcmp", [lData; rData])
            ArithConstantOp(zero32, 0L)
            ArithCmpIOp(boolResult, "eq", cmpResult, zero32)
        ]
        (boolResult, lops @ rops @ ops)
    else
        let result = { Name = freshName env; Type = I1 }
        (result, lops @ rops @ [ArithCmpIOp(result, "eq", lv, rv)])
```

Replace the `NotEqual` case similarly:

```fsharp
| NotEqual (lhs, rhs, _) ->
    let (lv, lops) = elaborateExpr env lhs
    let (rv, rops) = elaborateExpr env rhs
    if lv.Type = Ptr then
        // String inequality via strcmp != 0
        let lDataPtr  = { Name = freshName env; Type = Ptr }
        let lData     = { Name = freshName env; Type = Ptr }
        let rDataPtr  = { Name = freshName env; Type = Ptr }
        let rData     = { Name = freshName env; Type = Ptr }
        let cmpResult = { Name = freshName env; Type = I32 }
        let zero32    = { Name = freshName env; Type = I32 }
        let boolResult = { Name = freshName env; Type = I1 }
        let ops = [
            LlvmGEPStructOp(lDataPtr, lv, 1)
            LlvmLoadOp(lData, lDataPtr)
            LlvmGEPStructOp(rDataPtr, rv, 1)
            LlvmLoadOp(rData, rDataPtr)
            LlvmCallOp(cmpResult, "@strcmp", [lData; rData])
            ArithConstantOp(zero32, 0L)
            ArithCmpIOp(boolResult, "ne", cmpResult, zero32)
        ]
        (boolResult, lops @ rops @ ops)
    else
        let result = { Name = freshName env; Type = I1 }
        (result, lops @ rops @ [ArithCmpIOp(result, "ne", lv, rv)])
```

CRITICAL: `zero32` must be `ArithConstantOp(zero32, 0L)` where `zero32.Type = I32` — the
`ArithCmpIOp` compares two values of the SAME type (`I32`), not I64. The Printer emits
`arith.cmpi eq, %cmpResult, %zero32 : i32` — the type suffix matches `cmpResult.Type` (I32).

Verify the Printer handles `ArithCmpIOp` with I32 operands: look at the printOp case for
`ArithCmpIOp` — it uses `printType lhs.Type` as the type suffix. This correctly handles I32.

`elaborateExpr`, placed BEFORE the existing `App(Var("print"), ...)` cases.

**string_concat builtin:**

The AST for `string_concat "foo" "bar"` is `App(App(Var("string_concat"), String("foo")), String("bar"))`.
Match the doubly-nested App:

```fsharp
| App (App (Var ("string_concat", _), aExpr, _), bExpr, _) ->
    let (aVal, aOps) = elaborateExpr env aExpr
    let (bVal, bOps) = elaborateExpr env bExpr
    let result = { Name = freshName env; Type = Ptr }
    (result, aOps @ bOps @ [LlvmCallOp(result, "@lang_string_concat", [aVal; bVal])])
```

**to_string builtin:**

The AST for `to_string 42` is `App(Var("to_string"), Number(42))`.
Elaborate the argument first, then dispatch on the result type:

```fsharp
| App (Var ("to_string", _), argExpr, _) ->
    let (argVal, argOps) = elaborateExpr env argExpr
    let result = { Name = freshName env; Type = Ptr }
    let callee =
        match argVal.Type with
        | I1  -> "@lang_to_string_bool"
        | _   -> "@lang_to_string_int"   // I64 and any other numeric
    (result, argOps @ [LlvmCallOp(result, callee, [argVal])])
```

ORDERING CONSTRAINT: Place these cases in this order within `elaborateExpr`:
1. `App(Var("string_length"), ...)` — from plan 08-01 (already present)
2. `App(App(Var("string_concat"), ...), ...)` — new, must be BEFORE general App
3. `App(Var("to_string"), ...)` — new, must be BEFORE general App
4. `App(Var("print"), String(s), ...)` — existing static literal fast path
5. `App(Var("println"), String(s), ...)` — existing static literal fast path
6. `App(Var("print"), strExpr, ...)` — general print from plan 08-01
7. `App(Var("println"), strExpr, ...)` — general println from plan 08-01
8. `App(funcExpr, argExpr, ...)` — general App dispatch

The `App(App(...), ...)` pattern for string_concat must appear BEFORE the general `App` case
because F# pattern matching is top-to-bottom and the general `App` case would otherwise
catch it first.

No changes to `elaborateModule` needed — `@lang_string_concat`, `@lang_to_string_int`, and
`@lang_to_string_bool` were declared in the externalFuncs list in plan 08-01.
