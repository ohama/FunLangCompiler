# Milestone v21.0: Partial Env Pattern

**Status:** SHIPPED 2026-04-02
**Phases:** 65
**Total Plans:** 2

## Overview

v21.0 completes Issue #5 by moving env allocation and capture storage to the definition site, so that LetRec body calls to 3+ arg curried functions find their captures already in the env. This is a single focused change to Elaboration.fs with regression verification against all 246 existing tests.

## Phases

### Phase 65: Partial Env Pattern Implementation

**Goal**: LetRec body에서 3+ arg curried function + outer capture 호출이 crash 없이 올바른 값을 반환한다
**Depends on**: Phase 64 (v20.0 caller-side env)
**Requirements**: ENV-01, ENV-02, ENV-03, REC-01, REC-02, REG-01, REG-02, REG-03, TEST-01, TEST-02
**Plans**: 2 plans

Plans:
- [x] 65-01: Definition site env allocation + immediate capture store (ENV-01~03, REC-01~02)
- [x] 65-02: Regression verification + E2E tests (REG-01~03, TEST-01~02)

**Success Criteria** (all verified):
  1. Definition site에서 GC_malloc'd env에 captures가 즉시 저장
  2. LetRec body에서 3+ arg curried function 올바른 결과값 반환
  3. 기존 2-arg curried function/capture 없는 curried function 동일 동작
  4. 248개 E2E 테스트 전부 통과 (246 기존 + 2 신규)

## Milestone Summary

**Key Decisions:**
- Template env stored in LLVM mutable global @__tenv_<name> (LetRec body func.funcs cannot reference @main SSA values)
- Template-copy fallback returns Ptr closure (not IndirectCallOp) — mirrors maker behavior
- Template-copy (no mutation) — fresh env per call to avoid corruption of shared template
- TplGlobals as ref list in ElabEnv for cross-func-boundary shared state

**Issues Resolved:**
- Issue #5: LetRec body에서 3+ arg curried function + outer capture 호출 시 crash — definition-site partial env pattern으로 완전 해결

**Issues Deferred:**
- None

**Technical Debt Incurred:**
- MutablePtrGlobal added to MlirIR (minimal, well-contained)
