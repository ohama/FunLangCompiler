  <action>
Add three cases to the `elaborateExpr` match expression. Place them after the `Divide` / `Modulo` cases and before the `Negate` case (or anywhere before the catch-all, order does not affect correctness):

**PipeRight: x |> f  ≡  f x  ≡  App(right, left)**
    | PipeRight (left, right, span) ->
        elaborateExpr env (App(right, left, span))

**ComposeRight: f >> g  ≡  fun x -> g (f x)**
    | ComposeRight (f, g, span) ->
        let n = env.Counter.Value
        env.Counter.Value <- n + 1
        let paramName = sprintf "__comp_%d" n
        let innerApp = App(f, Var(paramName, span), span)
        let outerApp = App(g, innerApp, span)
        elaborateExpr env (Lambda(paramName, outerApp, span))

**ComposeLeft: f << g  ≡  fun x -> f (g x)**
    | ComposeLeft (f, g, span) ->
        let n = env.Counter.Value
        env.Counter.Value <- n + 1
        let paramName = sprintf "__comp_%d" n
        let innerApp = App(g, Var(paramName, span), span)
        let outerApp = App(f, innerApp, span)
        elaborateExpr env (Lambda(paramName, outerApp, span))

Note: `__comp_N` uses `env.Counter` (the SSA name counter) to guarantee uniqueness. The prefix `__comp_` is never produced by the LangThree parser so it cannot shadow user variables.

The `Lambda` case in `elaborateExpr` handles named lambdas that appear as values (closure-maker path). However, since ComposeRight/Left produce a Lambda that is immediately the result of an expression (not bound to a name via `Let`), it will hit the general `Lambda` case. Verify that the existing Lambda elaboration (or the Let+Lambda two-arg pattern) handles this correctly. If `Lambda` as a bare expression is not handled, add:
    | Lambda(param, body, _) ->
        // Lambda value: create a single-capture closure on the fly
        // For now, desugar as Let("__lam_N", Lambda(...), Var "__lam_N")
        // so it goes through the Let+Lambda closure-maker path.
        let n = env.Counter.Value
        env.Counter.Value <- n + 1
        let lamName = sprintf "__lam_%d" n
        let letExpr = Let(lamName, Lambda(param, body, Ast.unknownSpan), Var(lamName, Ast.unknownSpan), Ast.unknownSpan)
        elaborateExpr env letExpr

This is only needed if composing two named functions (e.g., `inc >> dbl`) causes issues. For the success criteria tests, `inc` and `dbl` are `LetRec`-bound and hence in `KnownFuncs`, so the compose lambda will use `App(Var "inc", ...)` which is a direct call — no bare Lambda value involved.
