  Add an `ensureAdtFieldTypes` function near `ensureConsFieldTypes` (line ~1062):
  ```fsharp
  let ensureAdtFieldTypes (scrutAcc: MatchCompiler.Accessor) (argAccs: MatchCompiler.Accessor list) : MlirOp list =
      // For arity >= 1: do NOT preload field 1 here.
      // Let resolveAccessor handle the type naturally:
      // - For int payloads (Some n): field 1 loads as I64 by default -> correct
      // - For tuple payloads (Pair(a,b)): the Ptr-retype guard in resolveAccessor handles it
      []
  ```
  Actually, the simplest correct approach: do NOT preload at all. Instead, rely on the Ptr-retype guard in resolveAccessor (see below). Add the AdtCtor case to preloadOps:
  ```fsharp
  | MatchCompiler.AdtCtor(_, _, arity) when arity > 0 -> ensureAdtFieldTypes scrutAcc argAccs
  | MatchCompiler.AdtCtor _ -> []
  ```

  **4. resolveAccessor Ptr-retype guard:**
  This is the critical fix for multi-arg (Pair) pattern matching. In `resolveAccessor`, in the `Field(parent, idx)` case (line ~924), after resolving the parent, add a guard:

  If `parentVal.Type <> Ptr`, then the parent was cached with wrong type for GEP (e.g., field 1 of ADT block was loaded as I64 but holds a tuple Ptr). Re-resolve the parent as Ptr using `resolveAccessorTyped parent Ptr`.

  ```fsharp
  | MatchCompiler.Field (parent, idx) ->
      let (rawParentVal, parentOps) = resolveAccessor parent
      // Guard: if parent is not Ptr, we can't GEP into it. Re-resolve as Ptr.
      let (parentVal, retypeOps) =
          if rawParentVal.Type = Ptr then (rawParentVal, [])
          else resolveAccessorTyped parent Ptr
      let slotVal  = { Name = freshName env; Type = Ptr }
      let gepOp    = LlvmGEPLinearOp(slotVal, parentVal, idx)
      let fieldVal = { Name = freshName env; Type = I64 }
      let loadOp   = LlvmLoadOp(fieldVal, slotVal)
      accessorCache.[acc] <- fieldVal
      (fieldVal, parentOps @ retypeOps @ [gepOp; loadOp])
  ```

  This ensures that when `Field(Field(scrutAcc,1), 0)` is resolved for `Pair(a,b)` pattern:
  - First resolve `Field(scrutAcc, 1)` -> cached as I64 (default)
  - Then resolve `Field(Field(scrutAcc,1), 0)` -> parent is I64, not Ptr
  - Guard triggers: re-resolve `Field(scrutAcc, 1)` as Ptr via resolveAccessorTyped
  - Now GEP field 0 of the Ptr (tuple pointer) -> load I64 -> correct

  For `Some n` pattern:
  - `Field(scrutAcc, 1)` resolves as I64 (default) -> `n` binds to I64 -> correct (int stored directly at field 1)

  **Verify the guard does not break existing tests:** The guard only triggers when `parentVal.Type <> Ptr`. For cons cells: field 0 is I64 (head, a leaf value, never GEP'd further), field 1 is Ptr (tail, pre-loaded by ensureConsFieldTypes). For tuples: all fields loaded as I64 by default, and tuple fields are leaves (not further destructured in current tests). So the guard should be inert for all existing patterns.
