  <name>Task 1: Fix closure capture for mutable variables</name>
  <files>src/LangBackend.Compiler/Elaboration.fs</files>
  <action>
  Two changes in the closure compilation code (the `Let(name, Lambda(outerParam, Lambda(innerParam, innerBody)), inExpr)` case, around line 338):

  1. **Propagate MutableVars to innerEnv** (line ~358-366):
  The innerEnv construction currently omits MutableVars. Add `MutableVars = env.MutableVars` to the record:
  ```fsharp
  let innerEnv : ElabEnv =
      { Vars = Map.ofList [(innerParam, arg1Val)]
        Counter = ref 0; LabelCounter = ref 0; Blocks = ref []
        KnownFuncs = env.KnownFuncs
        Funcs = env.Funcs
        ClosureCounter = env.ClosureCounter
        Globals = env.Globals
        GlobalCounter = env.GlobalCounter
        TypeEnv = env.TypeEnv; RecordEnv = env.RecordEnv; ExnTags = env.ExnTags
        MutableVars = env.MutableVars }
  ```
  This ensures that when the closure body executes `Var(name)` for a captured mutable, it correctly emits LlvmLoadOp through the captured pointer.

  2. **Make capture load type conditional on MutableVars** (line ~370-374):
  Currently all captures are loaded as I64:
  ```fsharp
  let capVal = { Name = sprintf "%%t%d" (i + numCaptures); Type = I64 }
  ```
  Change to:
  ```fsharp
  let capType = if Set.contains capName env.MutableVars then Ptr else I64
  let capVal = { Name = sprintf "%%t%d" (i + numCaptures); Type = capType }
  ```
  This ensures the closure env slot for a mutable capture holds a Ptr (the ref cell address), not an I64 value. The `LlvmLoadOp` in the capture load section will then emit `llvm.load %ptr : !llvm.ptr -> !llvm.ptr` (loading a pointer from the env slot), and the subsequent `Var` deref will load the I64 value from that pointer.

  **Why this works:** The capture store code (line ~427-443) already stores `env.Vars[capName]` which for mutable vars IS the cell Ptr. No change needed there. The capture load just needs the correct type annotation so MLIR type-checks.
  </action>
  <verify>Run `dotnet build` — must compile. Then manually test:
  Create a temp file with `let mut x = 5 in let f = fun _ -> x in let _ = x <- 99 in f 0` and compile+run — should exit with 99 (not 5).</verify>
  <done>Closure captures mutable variable's ref cell pointer. Closure body correctly dereferences through captured pointer. Mutations after closure creation are visible inside closure.</done>
