```fsharp
// Phase 22: array_create builtin -- C runtime allocation
| App (App (Var ("array_create", _), nExpr, _), defExpr, _) ->
    let (nVal, nOps)     = elaborateExpr env nExpr
    let (defVal, defOps) = elaborateExpr env defExpr
    let result = { Name = freshName env; Type = Ptr }
    // defVal may be Ptr (e.g., storing strings). lang_array_create takes (i64, i64).
    // If defVal.Type = Ptr, coerce with LlvmPtrToIntOp; if I64, pass directly.
    let coerceOps, coercedDefVal =
        if defVal.Type = Ptr then
            let coerced = { Name = freshName env; Type = I64 }
            ([LlvmPtrToIntOp(coerced, defVal)], coerced)
        else
            ([], defVal)
    (result, nOps @ defOps @ coerceOps @ [LlvmCallOp(result, "@lang_array_create", [nVal; coercedDefVal])])
```

// --- Input:
exception ArrayError of string

  try
    let a = array_create 3 0 in

  with
  | ArrayError msg -> 999
// --- Output:

IMPORTANT: The OOB exception thrown by `lang_array_bounds_check` via `lang_throw` passes a `LangString*` as the exception value. The `try/with` handler catches ANY longjmp-thrown exception. The `with` clause uses the match compiler to dispatch on the caught exception. Since the thrown value is a raw `LangString*` (not an ADT DataValue with a tag), the wildcard pattern `_` would be safest. However, the existing exception syntax requires `exception Foo of type` declarations and `| Foo x -> ...` patterns.

The actual catch mechanism: `lang_throw` stores the value in `lang_current_exception_val`, then `longjmp`s. The `with` clause reads `lang_current_exception()` and dispatches. Since the thrown value is a bare `LangString*` (not a tagged ADT), a declared exception pattern like `ArrayError msg` may not match (it expects a DataValue with tag field).

Use the wildcard catch pattern instead. Check test `19-06-try-fallthrough.flt` -- that test shows what happens when NO pattern matches (the exception re-raises). For a wildcard catch, use `| _ -> 999`. The exact syntax:

```
let _ =
  try
    let a = array_create 3 0 in
    array_get a 5
  with
  | _ -> 999
```

If the parser requires at least one exception declaration for `try/with`, declare a dummy one. Test by checking if `try expr with | _ -> fallback` compiles without an exception declaration. If not, add `exception Dummy of string` before it.

Expected output: `999` followed by `0` (the exit code on next line).
