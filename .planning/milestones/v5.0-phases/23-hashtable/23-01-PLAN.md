1. Constants and hash function:
```c
#define LANG_HT_INITIAL_CAPACITY 16
#define LANG_HT_LOAD_FACTOR_NUM  3
#define LANG_HT_LOAD_FACTOR_DEN  4

static uint64_t lang_ht_hash(int64_t key) {
    uint64_t h = (uint64_t)key;
    h ^= h >> 33;
    h *= 0xff51afd7ed558ccdULL;
    h ^= h >> 33;
    h *= 0xc4ceb9fe1a85ec53ULL;
    h ^= h >> 33;
    return h;
}
```

2. Internal lookup helper:
```c
static LangHashEntry* lang_ht_find(LangHashtable* ht, int64_t key) {
    uint64_t h = lang_ht_hash(key) % (uint64_t)ht->capacity;
    LangHashEntry* e = ht->buckets[h];
    while (e != NULL) {
        if (e->key == key) return e;
        e = e->next;
    }
    return NULL;
}
```

3. `lang_hashtable_create`: Allocates LangHashtable with GC_malloc, capacity=16, size=0, buckets GC_malloc'd array (GC_malloc zero-initialises, so all bucket heads start NULL).

4. `lang_hashtable_get`: Calls lang_ht_find. If NULL, constructs LangString* error message "Hashtable.get: key not found" and calls `lang_throw((void*)msg)`. Otherwise returns `e->val`. Follow the exact pattern from 23-RESEARCH.md Pattern 2 / Code Examples section.

5. `lang_hashtable_set`: Check load factor (`size * LANG_HT_LOAD_FACTOR_DEN >= capacity * LANG_HT_LOAD_FACTOR_NUM`), rehash if exceeded (double capacity, re-bucket all entries). Then call lang_ht_find — if found, update `e->val`; if not, allocate new LangHashEntry with GC_malloc, prepend to bucket chain, increment size. Use `(uint64_t)` cast on capacity in modulo operations.

6. `lang_hashtable_containsKey`: Returns `lang_ht_find(ht, key) != NULL ? 1 : 0`.

7. `lang_hashtable_remove`: Walk bucket chain with prev-pointer technique. If found, unlink (`*prev = e->next`), decrement size. If not found, no-op (consistent with LangThree semantics).

8. `lang_hashtable_keys`: Iterate all buckets and all entries, building a LangCons* linked list. For each entry, GC_malloc a LangCons cell with `cell->head = e->key; cell->tail = head; head = cell`. Return head (NULL if empty table, which is the nil cons list).

IMPORTANT:
- ALL allocations must use GC_malloc (not malloc)
- Include `<string.h>` if not already included (for strlen/memcpy in error message construction)
- Use `(uint64_t)ht->capacity` in modulo to avoid signed division issues
- The `lang_hashtable_get` error path must use `lang_throw` exactly like `lang_array_bounds_check` does
