  <action>
Replace the `App` case in `elaborateExpr` with the 3-way dispatch from 05-RESEARCH.md Pattern 7. The new logic:

```fsharp
| App (funcExpr, argExpr, _) ->
    match funcExpr with
    | Var (name, _) ->
        match Map.tryFind name env.KnownFuncs with
        | Some sig_ when sig_.ClosureInfo.IsNone ->
            // CASE 1: DIRECT CALL (Phase 4 behavior — known non-closure function)
            let (argVal, argOps) = elaborateExpr env argExpr
            let result = { Name = freshName env; Type = sig_.ReturnType }
            (result, argOps @ [DirectCallOp(result, sig_.MlirName, [argVal])])
        | Some sig_ ->
            // CASE 2: CLOSURE-MAKING CALL — allocate env in current frame, then call
            let ci = sig_.ClosureInfo.Value
            let (argVal, argOps) = elaborateExpr env argExpr
            let countVal = { Name = freshName env; Type = I64 }
            let envPtrVal = { Name = freshName env; Type = Ptr }
            let resultVal = { Name = freshName env; Type = Ptr }
            let setupOps = [
                ArithConstantOp(countVal, 1L)
                LlvmAllocaOp(envPtrVal, countVal, ci.NumCaptures)
            ]
            let callOp = DirectCallOp(resultVal, sig_.MlirName, [argVal; envPtrVal])
            (resultVal, argOps @ setupOps @ [callOp])
        | None ->
            // CASE 3: Check Vars for closure value (type Ptr)
            match Map.tryFind name env.Vars with
            | Some closureVal when closureVal.Type = Ptr ->
                // INDIRECT/CLOSURE CALL
                let (argVal, argOps) = elaborateExpr env argExpr
                let fnPtrVal = { Name = freshName env; Type = Ptr }
                let result = { Name = freshName env; Type = I64 }
                let loadOp = LlvmLoadOp(fnPtrVal, closureVal)
                let callOp = IndirectCallOp(result, fnPtrVal, closureVal, argVal)
                (result, argOps @ [loadOp; callOp])
            | _ ->
                failwithf "Elaboration: unsupported App — '%s' is not a known function or closure value" name
    | _ ->
        failwithf "Elaboration: unsupported App (only named function application supported in Phase 5)"
```

**Key points:**
- Case 1 (direct call): Identical to Phase 4 behavior. `sig_.ClosureInfo.IsNone` check distinguishes from closure-makers.
- Case 2 (closure-making call): The CALLER allocates the env struct via `LlvmAllocaOp` in the current frame. The alloca count is always 1 (one struct). `numCaptures` determines the struct type `!llvm.struct<(ptr{, i64}^N)>`. Then calls the closure-maker via `DirectCallOp` with both the arg and the pre-allocated env pointer.
- Case 3 (indirect closure call): Loads fn_ptr from `closureVal` (which is the env pointer, with fn_ptr at offset 0) via `LlvmLoadOp`, then dispatches via `IndirectCallOp(result, fnPtr, envPtr, arg)`. The closureVal serves as both the env pointer and the source of the fn_ptr.

**CRITICAL:** The closure-making call returns `Ptr` type (the filled env pointer). When the result is bound via `Let`, the name maps to a `Ptr`-typed MlirValue in Vars. The subsequent `App` on that name hits Case 3.
