# Phase 38: CLI Arguments - Research

**Researched:** 2026-03-30
**Domain:** C runtime argv storage + MLIR @main signature change + `get_args` builtin elaboration
**Confidence:** HIGH

## Summary

Phase 38 requires two coordinated changes. First, the generated `@main` MLIR function must accept `(i64, !llvm.ptr)` parameters (argc and argv) so the OS can pass command-line arguments to the binary. Second, those arguments must be stored in the runtime at startup and made retrievable via a new `get_args ()` builtin that returns a `string list` (a `LangCons*` chain of `LangString*` pointers).

The current `@main` is generated by `elaborateProgram` in `Elaboration.fs` with `InputTypes = []`. To add parameters, `InputTypes` must become `[I64; Ptr]` and the entry block of `@main` must begin with a `LlvmCallVoidOp("@lang_init_args", [arg0Val; arg1Val])` — where `arg0Val` and `arg1Val` are the MLIR block-argument values `%arg0 : i64` and `%arg1 : !llvm.ptr`. These block arguments are represented as `MlirValue` records in the entry block's `Args` list, but since the entry block has no label (`Label = None`), they cannot be expressed as block arguments in the current `MlirBlock` model. The correct solution is to declare them as function parameters and access them via MLIR's `%arg0`/`%arg1` naming convention — which the `Printer` already emits for `func.func`. After lowering to LLVM IR, `%arg0`/`%arg1` become the C-ABI `argc`/`argv` values automatically.

The runtime change adds two C functions: `lang_init_args(int64_t argc, char** argv)` (called once at startup) and `lang_get_args()` (called from `get_args ()` builtin) which builds a `LangCons*` list of `LangString*` from the stored `argc`/`argv`. The builtin elaboration pattern matches `stdin_read_line` exactly — a unit-arg call that returns `Ptr`.

**Primary recommendation:** Add `lang_init_args`/`lang_get_args` to `lang_runtime.c/h`; change `@main InputTypes` to `[I64; Ptr]`; prepend `lang_init_args(%arg0, %arg1)` before `GC_init` in the entry block; add `get_args` builtin arm in `Elaboration.fs`; add ExternalFuncDecl entries to both `externalFuncs` lists; write E2E test.

## Architecture Patterns

### Pattern 1: `@main` with `(i64, ptr)` Parameters

**What:** Change `FuncOp.InputTypes` to `[I64; Ptr]` so MLIR emits `func.func @main(%arg0: i64, %arg1: !llvm.ptr) -> i64`. After lowering via `--convert-func-to-llvm`, `@main` becomes `int main(int argc, char** argv)` in the generated C ABI.

**When to use:** Always — `@main` is the sole entry point for compiled binaries.

**Example:**
```fsharp
// Source: Elaboration.fs — elaborateProgram, updated mainFunc construction
let mainFunc : FuncOp =
    { Name        = "@main"
      InputTypes  = [I64; Ptr]   // argc: i64, argv: !llvm.ptr
      ReturnType  = Some resultVal.Type
      Body        = { Blocks = allBlocksWithGCAndArgs }
      IsLlvmFunc  = false }
```

The `Printer.printFuncOp` will emit:
```
func.func @main(%arg0: i64, %arg1: !llvm.ptr) -> i64 {
```

### Pattern 2: Accessing `%arg0`/`%arg1` in the Entry Block

**What:** `func.func` parameters are named `%arg0`, `%arg1`, ... automatically by the printer (line 174 of `Printer.fs`). In the IR, they are accessible as `MlirValue` records without any explicit allocation.

**When to use:** When a call inside the entry block needs to reference the function's input parameters.

**Example:**
```fsharp
// Source: Elaboration.fs — construct arg values for lang_init_args call
let argcVal = { Name = "%arg0"; Type = I64 }
let argvVal = { Name = "%arg1"; Type = Ptr }
let initArgsOp = LlvmCallVoidOp("@lang_init_args", [argcVal; argvVal])
```

These hard-coded names are safe because `%arg0` and `%arg1` are the names the Printer always generates for input types at index 0 and 1.

### Pattern 3: Prepending `lang_init_args` Before `GC_init`

**What:** The entry block body is currently `gcInitOp :: entryOps`. After this change it becomes `initArgsOp :: gcInitOp :: entryOps`. `lang_init_args` must be called before any user code that might call `get_args`, and before GC_init is also fine since `lang_init_args` only stores pointers in static globals without allocating.

**When to use:** Always — init args before user code runs.

**Example:**
```fsharp
// Source: Elaboration.fs — allBlocksWithGCAndArgs construction
let initArgsOp = LlvmCallVoidOp("@lang_init_args", [argcVal; argvVal])
let gcInitOp   = LlvmCallVoidOp("@GC_init", [])
let allBlocksWithGCAndArgs =
    match allBlocks with
    | [] -> allBlocks
    | entryBlock :: rest ->
        { entryBlock with Body = initArgsOp :: gcInitOp :: entryBlock.Body } :: rest
```

### Pattern 4: `get_args` Builtin Arm in `elaborateExpr`

**What:** Follows the identical pattern as `stdin_read_all` (unit-arg, returns Ptr). The unit expression is elaborated and discarded; a `LlvmCallOp` with no arguments returns a `Ptr`.

**When to use:** When the user writes `get_args ()` in their code.

**Example:**
```fsharp
// Source: Elaboration.fs — elaborateExpr, new arm (mirrors stdin_read_all pattern)
| App (Var ("get_args", _), unitExpr, _) ->
    let (_uVal, uOps) = elaborateExpr env unitExpr
    let result = { Name = freshName env; Type = Ptr }
    (result, uOps @ [LlvmCallOp(result, "@lang_get_args", [])])
```

### Pattern 5: `lang_get_args` C Implementation

**What:** Reads the stored `argc`/`argv` static globals and builds a `LangCons*` list of `LangString*`. Each `argv[i]` becomes a `LangString` allocated with `GC_malloc`. The `LangCons.head` stores the `LangString*` as `int64_t` (ptrtoint), matching the existing pattern used in `lang_hashtable_keys_str` and `lang_read_lines`.

**When to use:** Called once per `get_args ()` invocation. Returns a fresh list each call (stateless construction from static storage).

**Example:**
```c
// Source: lang_runtime.c — Phase 38 implementation
static int64_t  s_argc = 0;
static char**   s_argv = NULL;

void lang_init_args(int64_t argc, char** argv) {
    s_argc = argc;
    s_argv = argv;
}

LangCons* lang_get_args(void) {
    LangCons* head = NULL;
    // Build list in reverse then reverse it (or build forward with tail pointer)
    // Forward build with tail pointer is cleaner:
    LangCons** cursor = &head;
    for (int64_t i = 0; i < s_argc; i++) {
        int64_t len = (int64_t)strlen(s_argv[i]);
        char* buf = (char*)GC_malloc((size_t)(len + 1));
        memcpy(buf, s_argv[i], (size_t)(len + 1));
        LangString* s = (LangString*)GC_malloc(sizeof(LangString));
        s->length = len;
        s->data = buf;
        LangCons* cell = (LangCons*)GC_malloc(sizeof(LangCons));
        cell->head = (int64_t)(uintptr_t)s;
        cell->tail = NULL;
        *cursor = cell;
        cursor = &cell->tail;
    }
    return head;
}
```

### Anti-Patterns to Avoid

- **Using `MlirBlock.Args` for entry block parameters:** Block args are for labeled blocks (CFG phi-style). The entry block has no label and no block args. Function parameters are accessed via `%arg0`/`%arg1` naming, not block arg lists.
- **Calling `lang_get_args` with argv stored as raw `char**` in I64:** The `s_argv` field stays as `char**` in C — there is no need to store it as `int64_t` in the runtime. It is a C-side concern only.
- **Building the list in reverse without reversing:** `lang_read_lines` builds lines in order using the cursor pattern. Use the same cursor pattern to preserve argument order (argv[0] first in list).
- **Omitting `lang_init_args` from ExternalFuncDecl:** `lang_init_args` takes `(i64, ptr)` — it must be declared in both `externalFuncs` lists or MLIR validation will fail at call site.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| C string to `LangString*` | Custom per-phase logic | Reuse `strlen`/`memcpy`/`GC_malloc` pattern already in `lang_to_string_int` and `lang_file_read` | The pattern is identical: measure length, alloc buffer, copy, alloc struct, set fields |
| List construction in C | Reverse + re-reverse | Forward cursor pattern from `lang_range`/`lang_read_lines` | Single pass, correct order, no extra allocation |

**Key insight:** The `LangCons` struct, `LangString` struct, `GC_malloc`, and the ptr-as-i64 head convention are all established patterns. Phase 38 assembles them — it builds nothing new at the C level.

## Common Pitfalls

### Pitfall 1: `@main` Return Type After Adding Input Types
**What goes wrong:** `@main` currently has `InputTypes = []`. If changed to `[I64; Ptr]` but `ReturnType` is left as `Some I64`, the generated MLIR becomes `func.func @main(%arg0: i64, %arg1: !llvm.ptr) -> i64`. After `--convert-func-to-llvm`, this becomes `define i64 @main(i64 %arg0, ptr %arg1)`. The OS expects `int main(int argc, char** argv)` where the return type is `i32`. The existing `ReturnType = Some I64` was already used before this phase, so the binary already returns `i64`. This is an existing issue, not introduced by Phase 38. Do not change `ReturnType` in this phase — it is out of scope.
**How to avoid:** Keep `ReturnType = Some resultVal.Type` unchanged. The existing behavior is preserved.
**Warning signs:** N/A — this pre-existed Phase 38.

### Pitfall 2: Hard-Coded `%arg0`/`%arg1` Names Must Match Printer Output
**What goes wrong:** If `InputTypes = [I64; Ptr]`, the Printer emits `%arg0: i64, %arg1: !llvm.ptr`. The `LlvmCallVoidOp` for `lang_init_args` references `{ Name = "%arg0"; Type = I64 }` and `{ Name = "%arg1"; Type = Ptr }`. If the Printer ever changes its naming convention, these hard-coded names will silently generate broken MLIR.
**Why it happens:** `func.func` parameters are not represented as `MlirValue` objects in the IR data structures — they are implicitly named by the Printer.
**How to avoid:** The Printer's naming is deterministic (`%arg{i}` for index `i`) and has been stable across all 38 phases. This is safe. Add a comment in `Elaboration.fs` noting the dependency.
**Warning signs:** MLIR validation error "use of undefined value '%arg0'" if the Printer changes.

### Pitfall 3: ExternalFuncDecl Not Updated in Both Lists
**What goes wrong:** `Elaboration.fs` has two identical `externalFuncs` list literals (one in the module-path branch around line 3347, one in the program-path branch around line 3598). Adding `@lang_init_args` and `@lang_get_args` to only one list causes MLIR validation failures in the other path.
**Why it happens:** Two-list structure from module-aware elaboration. Phase 37 research documents this pattern.
**How to avoid:** Update both lists. Search for `@lang_get_cwd` (which was also added in Phase 27) and add the new entries immediately after it in both lists.
**Warning signs:** Test passes for simple programs, fails for module-using programs.

### Pitfall 4: `lang_init_args` Signature in MLIR vs C
**What goes wrong:** In MLIR, `@lang_init_args` takes `(i64, !llvm.ptr)`. In C, `argv` is `char**`. After `--mlir-to-llvmir`, `!llvm.ptr` becomes `ptr` (LLVM opaque pointer), which is the same as `char**` at the ABI level. No cast is needed. The C function signature `void lang_init_args(int64_t argc, char** argv)` is ABI-compatible with the MLIR declaration.
**How to avoid:** Declare in C as `void lang_init_args(int64_t argc, char** argv)`. Declare in MLIR ExternalFuncDecl as `ExtParams = [I64; Ptr]`. They match.
**Warning signs:** None expected — this is standard LLVM opaque pointer ABI.

### Pitfall 5: `get_args` Returns `Ptr` (a `LangCons*`), Not a List Wrapper
**What goes wrong:** The return value of `@lang_get_args` is a raw `LangCons*` typed as `Ptr` in MLIR. When user code iterates with `for x in get_args () do`, the for-in elaborator must recognize this as a list. This works because `lang_for_in_list` already handles `LangCons*` — the type inference for collection kind is done at elaboration time based on the call expression pattern. Since `get_args ()` is a `LlvmCallOp` returning `Ptr`, the `for-in` arm treats it as a generic collection and calls `lang_for_in` (the dispatcher), which works for `LangCons*` because `lang_for_in` checks the tag field — and for a `LangCons*`, the head field is non-null (or null for empty), not a tag sentinel.
**Why it happens:** `lang_for_in` dispatches on the first `int64_t` word of the collection pointer. For arrays, this is a non-negative length. For hashtables, it is -1 or -2. For cons lists, the first word is the first element's `int64_t head` value (could be 0 for an empty list).
**How to avoid:** Use `lang_for_in_list` directly (as done for `read_lines` results). The collection returned by `get_args ()` is a `LangCons*` list and iterates correctly with `lang_for_in_list`. The elaborator's `CollectionKind` inference must recognize the `get_args ()` call as a list. Check whether the elaborator's `inferCollectionKind` or equivalent recognizes builtin-returning-list calls — if not, `for x in get_args ()` may call `lang_for_in` (generic) instead of `lang_for_in_list`. Test this in the E2E test.
**Warning signs:** `for x in get_args () do` crashes or skips elements.

## Open Questions

1. **`for x in get_args () do` — collection kind dispatch**
   - What we know: `lang_for_in` dispatches on the first word of the collection. For a `LangCons*`, the first word is `head` (an `int64_t`). If `head == 0` (empty list or first arg is null string pointer), `lang_for_in` might misidentify it. The dedicated `lang_for_in_list` is safe for cons lists. The elaborator uses `inferCollectionKind` (around line 60 in `Elaboration.fs`) to select the right for-in function.
   - What's unclear: Whether `inferCollectionKind` can identify a `LlvmCallOp` returning a `LangCons*` as a list vs generic collection.
   - Recommendation: Test `for x in get_args () do println x` in the E2E test. If it fails, add a `CollectionKind` for `get_args` return values, or wrap `get_args ()` in a let binding before iterating.

2. **argv[0] inclusion in `get_args` list**
   - What we know: C's `argv[0]` is the program name, not a user-supplied argument. FunLexYacc likely wants only the user args (argv[1..argc-1]).
   - What's unclear: The Phase 38 success criteria says `./prog foo bar` should allow printing each argument — it is ambiguous whether `./prog` itself is included.
   - Recommendation: Include all arguments starting from `argv[0]` (i.e., iterate `i = 0` to `argc-1`). This matches the behavior of `System.Environment.GetCommandLineArgs()` in .NET (which includes the binary name). The E2E test can verify which args are returned. If FunLexYacc needs argv[1..], start from `i = 1`. The success criterion says "a compiled binary invoked with `./prog foo bar` can print each argument" — this suggests printing `foo` and `bar`, not `./prog`. **Recommend starting from `i = 1`** (skip argv[0]).

3. **Duplicate `@main InputTypes` pattern for module-path elaboration**
   - What we know: There are two `mainFunc` constructions in `Elaboration.fs` (around lines 3340 and 3591). Both must be updated to `InputTypes = [I64; Ptr]`. Both need the `initArgsOp :: gcInitOp` prefix.
   - What's unclear: Whether both paths are tested by the existing E2E test suite.
   - Recommendation: Update both `mainFunc` constructions. Search for `InputTypes  = []` near `Name = "@main"` in `Elaboration.fs` to find both.
