  <name>Task 3: Verify caller-side indirect calls still work</name>
  <files>src/FunLangCompiler.Compiler/Elaboration.fs</files>
  <action>
  Check the 4 IndirectCallOp call sites (lines ~2579, ~2589, ~2614, ~2623) to verify they work correctly with the new ABI.

  At line ~2579 and ~2614: `IndirectCallOp(result, fnPtrVal, closureVal, argVal)` — the Printer emits `(!llvm.ptr, ARG_TYPE) -> i64` as the call type. If argVal is Ptr, the call type matches the new closure signature. If argVal is I64, the call says `(!llvm.ptr, i64)` but the function actually is `(!llvm.ptr, !llvm.ptr)`.

  In LLVM dialect MLIR, indirect calls through opaque `!llvm.ptr` function pointers do NOT cross-check the call type against the function definition. The type annotation on the indirect call is only used locally for the calling convention. Since i64 and ptr are both 64-bit, this is safe at the machine level.

  However, if any existing tests fail due to caller-side type mismatch, add inttoptr coercion for i64 args at the indirect call sites:
  ```fsharp

      if argVal.Type = I64 then
          let ptrVal = { Name = freshName env; Type = Ptr }
          (ptrVal, [LlvmIntToPtrOp(ptrVal, argVal)])
      else
          (argVal, [])
  ```

  Only add this coercion IF tests fail without it. Do NOT add it preemptively.
