3. Update `LangCons` struct: add `int64_t heap_tag` as FIRST field.
   ```c
   typedef struct LangCons {
       int64_t         heap_tag;
       int64_t         head;
       struct LangCons* tail;
   } LangCons;
   ```
4. Every site that creates a `LangCons*` via `GC_malloc(sizeof(LangCons))` must set `cell->heap_tag = LANG_HEAP_TAG_LIST`. Search for all `GC_malloc(sizeof(LangCons))` sites (~15+). The existing `cell->head = ...` and `cell->tail = ...` remain correct since field names didn't change.

5. Update `lang_hashtable_trygetvalue`: currently allocates `GC_malloc(16)` for a 2-slot tuple. Change to allocate `(2+2)*8 = 32` bytes, set slot 0 = LANG_HEAP_TAG_TUPLE, slot 1 = 2 (field count), slot 2 = tagged bool, slot 3 = value.
   ```c
   int64_t* tup = (int64_t*)GC_malloc(32);
   tup[0] = LANG_HEAP_TAG_TUPLE;
   tup[1] = 2;  // field count
   // ... existing tup[0]/tup[1] values move to tup[2]/tup[3]
   ```

7. Do NOT change: arrays (length at slot 0), hashtables (tag=-1), closures, mutable cells, StringBuilder, HashSet, Queue, MutableList, LangHashEntry, LangQueueNode, etc.

**Important:** Since C struct access uses named fields (`s->length`, `cell->head`), most existing C code that reads/writes these fields will automatically work after adding the `heap_tag` field -- the compiler offsets based on field name, not position. The only manual-offset code (`tup[0]`, `tup[1]`) is in `lang_hashtable_trygetvalue` and `lang_for_in_hashtable`.
